## The Problem

1.4 gave you the cost/capability axis: Haiku, Sonnet, and Opus are the same request/response shape from 1.3, priced and scaled differently, and picking between them is a real per-task engineering decision, not a default you leave untouched. That completes the mental model this module set out to build: sampling (1.1), the token/context ceiling (1.2), the request/response shape (1.3), and now which model tier to send that request to (1.4).

Here's the last piece, and it's not a new idea — it's a naming exercise. LangChain is a popular framework that wraps exactly the model you now have in your head. If you've seen `Chain`, `AgentExecutor`, or `PromptTemplate` in a job posting or a production codebase and treated them as their own thing to learn, this section's job is to convince you they aren't. They're names for pieces you already understand, glued together in a library.

## The Concept

A framework like LangChain doesn't add a new capability on top of the raw API. It gives you pre-built classes for patterns you'd otherwise write by hand: building a prompt from a template and some variables, chaining several model calls together, running the tool-call-then-feed-result-back loop without writing the loop yourself, and swapping which model provider you're talking to without rewriting your calling code. Every one of those patterns is something Module 1 already gave you the vocabulary for. LangChain's value is convenience and a large library of pre-built integrations (connectors to vector databases, document loaders, output parsers) — not a different way requests and responses actually work underneath.

That's the whole point of naming it here, at the end of a module about how LLMs actually work: once you know what's really in a request and a response, a framework's class names stop being magic words and start being labels on familiar furniture.

### The abstractions, mapped onto what you already know

**`PromptTemplate` → 1.3's request text, with blanks.** In 1.3, you saw that a request is fundamentally a system prompt plus a sequence of messages, sent as text (plus a tool schema, once Module 3 adds tools). A `PromptTemplate` is just a string with `{placeholders}` in it — "Summarize this ticket: `{ticket_text}`, in the tone of `{persona}`" — plus code that fills those placeholders in and hands you back the exact string 1.3 already described. It doesn't change what gets sent to the model. It changes how you *build* the string before sending it, which matters once you have dozens of prompts across a codebase and don't want to hand-concatenate strings everywhere.

**`Chain` → a fixed sequence of request/response round trips, glued together.** The simplest chain is one call: fill a template, send it, get a response back — the same round trip from 1.3, just wrapped in an object with a `.run()` method. A more useful chain strings several of those together: call model A to extract structured facts from a document, feed those facts into a second prompt, call model B (maybe a cheaper tier, per 1.4's tradeoff) to generate a summary from them. Nothing here is a new primitive. It's the same "send a request, get a response, do something with it" loop from 1.3, run more than once in a row, with the output of one round trip feeding the input of the next. LangChain gives this a class name and handles the plumbing of passing outputs to inputs; you could write the same thing yourself as two function calls in sequence.

**`AgentExecutor` → the tool-use loop, wrapped in a class.** This is the one worth being precise about, because Module 3 builds it by hand and it's the piece with the most going on underneath. When Claude (or any tool-calling model) decides it needs a tool, the response comes back with a tool-call request instead of a final answer. Something then has to: parse which tool was requested and with what arguments, actually execute that tool, take the result, and send it back to the model as part of the next request in the same conversation — repeating until the model produces a final answer instead of another tool call. That parse-execute-feed-back-repeat loop is exactly what Module 3.3 builds from scratch. `AgentExecutor` is LangChain's name for the same loop, pre-written: you register your tools with it, and it runs that loop for you instead of you writing the `while` loop and the branch on response type yourself.

**LangChain's model-provider interface → a coincidence to notice now.** Part of why LangChain is popular is that it defines one interface for "a chat model" and then has separate implementations behind it for Claude, GPT, Gemini, and others — swap the constructor, keep the rest of your chain or agent code unchanged. That's useful in a codebase that genuinely needs to support more than one provider. It's also, not coincidentally, exactly the shape of a decision Module 9 walks you through building yourself: a thin `ModelProvider` interface, used for one bounded swap (one specialist agent onto a different provider, evaluated against the same benchmark), rather than rearchitecting the whole course's Claude-specific depth around provider-agnosticism from day one. LangChain solves that same problem at framework scale, for every chain and agent it ships; Module 9 solves a narrower, honestly-scoped version of it, once you've already felt why a Claude-only response shape (Module 2.4's deep dive into `stop_reason`, usage, content blocks) is worth having before you abstract it away.

**LangGraph → `AgentExecutor`, generalized to more than one agent.** LangChain's own team built a second framework, LangGraph, specifically because `AgentExecutor`'s loop assumes a single agent looping on its own tool calls — it doesn't have a native way to express "agent A hands off to agent B" or "route this task to one of several specialists based on what it is." LangGraph models an agentic system as a graph: nodes are steps (which can be individual model calls or whole agents), edges are the possible transitions between them, and a shared state object flows through the graph as it executes. That's not a new idea either, from where you'll be standing by Module 9 — it's the same fleet-plus-orchestrator shape Module 8 builds by hand (a shared agent loop, reused per specialist) and Module 9 wires together with routing logic and a shared `OrchestrationState`. Module 9.8 comes back to this comparison directly, once you've built the orchestrator yourself and have something concrete to compare LangGraph's graph model against.

None of the above is an argument that frameworks are bad. LangChain buys you two real things: faster prototyping (a working agent loop in a few lines instead of a hand-written one), and a large ecosystem of pre-built integrations — document loaders, vector store connectors, output parsers — that would otherwise be your own plumbing to write and maintain.

What it costs you is a layer of abstraction sitting between you and the actual request/response round trip. Under normal operation you don't notice it. You notice it the moment something goes wrong: a tool call comes back malformed and doesn't match the schema you defined, the model stops for a reason your code didn't anticipate, a conversation grows past the context window and errors out (1.2's hard ceiling, arriving as a runtime failure instead of a design constraint you planned around). When that happens through a framework, you're debugging *through* `AgentExecutor` — reading its source, or its error messages, or its GitHub issues — to figure out what actually happened at the raw request/response level, rather than reading a stack trace that points straight at your own loop.

That's the real reason this course builds these primitives by hand first, stated plainly rather than left implicit: by the time you reach for LangChain or LangGraph professionally — and you likely will, since a lot of production code and job postings assume familiarity with them — you'll already know exactly which raw primitive each class name is standing in for. A malformed tool call won't be a framework mystery; it'll be a familiar failure in the parse step of a loop you've written yourself. That's the difference between treating a framework's behavior as unexplainable magic and treating it as a convenience wrapper over a mental model you already own.

## What You Can Show Now

There's no code to demo — that's still one module away — but here's the full model this module built, piece by piece, worth holding as one picture before Module 2 makes it real:

- **1.1** — generation is next-token prediction, and sampling settings like temperature control how sharply the model commits to its most likely next token versus spreading weight across plausible alternatives. Not "distance" from your prompt — the shape of a probability distribution.
- **1.2** — a token is the actual unit the model reads and counts, and the context window is a hard ceiling, not a soft "memory" limit — a consequence of attention cost growing with the square of the conversation length, not an arbitrary product decision. Every later design choice that prunes or threads context (Module 6.9, Module 9.5) is working around this exact constraint.
- **1.3** — a request is a system prompt plus a message history (plus, from Module 3 on, tool definitions); a response is text plus metadata about why the model stopped and what it cost. That shape is what everything else in this course sends and receives.
- **1.4** — Haiku, Sonnet, and Opus are the same shape at different price and capability points, and choosing between them per task is a real design decision, not a default.
- **1.5** — frameworks like LangChain package all four of the above into named classes. Convenient, worth recognizing, and not a different mental model underneath.
